# Requirement Delivery Flow（从需求到交付的执行纪律：想清楚 → 最多确认一次 → 一口气做完 → 自检交付）

补上读写闭环中间那段「干活」的空白：`project-context-sync` 管开工读、`ai-handoff-doc-update` 管收尾写、`git-commit-guard` 管提交——但**从"收到需求"到"交付结果"这一段的执行节奏一直没人管**，于是 AI 容易用"继续推进/这就去做/不再问你"这类**空转播报**填满它，看着很忙，却既没结合代码认真思考，也没把需求真正落地。本 Skill 就是这段的纪律。

```text
开工：project-context-sync（读）
  ↓
【本 Skill】想清楚 → 最多确认一次 → 一口气实现 → 自检   ← 补上的这段
  ↓
收尾：ai-handoff-doc-update（写） → git-commit-guard（提交）
```

## 最重要的认知边界

- **状态播报不是进度。** "继续推进""马上做""不再问你了""正在实现中"——这些话**本身就是打断**，不产出任何价值。用户连发几次"继续"，说明他看到的全是保证、没有产出。真正的工作是**安静地思考 + 落地编辑**，进度靠"改了哪个文件、发现了什么"体现，不靠表决心。
- **"不空转"不等于"莽撞"。** 一口气做完的前提是**动手前已经想清楚**；撞上不可逆高风险动作或与方案冲突的新事实，仍要停下——刹车不是绕弯子。
- **"确认一次"不等于"从不确认"，也不等于"每步都确认"。** 需求明确 → 零确认直接做；把握不准/影响大 → 把可预见的取舍**一次性**讲清并确认，之后不再逐步请示。两个极端都错。
- **静态自检 ≠ 编译验证。** 只读代码看不出返回类型退化、重载二义、过时 API——所以自检默认 L0，遇到编译期才能暴露的风险要主动申请仅 compile 的 L1（沿用框架既有原则）。

## 红线

1. **不用状态播报代替工作（本 Skill 头号红线）。** 禁止发"继续推进/这就去做/不再问你/正在实现"这类无实质进展的话。进度更新只在**有实质产出或方向改变**时给，且必须带具体内容（改了哪个文件、发现了什么坑、方案为何调整）。
2. **确认最多一次，且必须是"完整方案"。** 需求明确时零确认。需要确认时，把「要改什么 + 关键取舍 + 影响面 + 待你拍板的点」一次性讲清一并确认，不搞"我可以开始了吗""下一步做 X 行不行"式的逐步请示。
3. **想清楚再动手。** 未结合当前代码理解现状（现有实现、可复用旧代码、影响面、角色口径）就开写，属于违规——绕弯子的根因往往是"没想清就动手，边写边改方向"。
4. **必须停下的例外（不计入"绕弯子"）：** ① 不可逆/高风险动作——改表结构、删数据、改公共方法、动权限、跨模块大范围改动；② 需求自相矛盾、无法两全；③ 实现中发现与已确认方案冲突的新事实。遇到这三类**停下外显**（冲突点 + 选择 + 理由），不闷头按错误方案做到底。
5. **不擅自构建/启动/写库。** 自检默认静态；构建（哪怕仅 compile）、跑服务、写库都需用户放行（沿用 `git-commit-guard`/`mysql-*` 红线）。

## 执行流程

### 第 1 步：想清楚（动手前的静态思考，先不播报）

结合当前代码，把下面几件事在脑子里过清楚——**这一步没有"状态播报"，只有阅读和思考**：

```text
现状        这个需求落在哪个模块？现有是怎么实现的？
复用        有没有可参照的旧代码/组件/接口/写法？（参照现有风格，不重复造轮子）
影响面      改动波及本模块哪些地方、哪些上下游调用方？是否动公共方法/表结构/字典/权限？
角色口径    功能给谁用？各角色能做什么、不能做什么？
```

想不清 = 还没到动手的时候。想清了 → 判断要不要确认。

### 第 2 步：判断要不要确认（最多一次）

```text
需求明确 + 影响可控                       → 不确认，直接进第 3 步
把握不准 / 影响面大（表结构/公共方法/权限/跨模块）→ 一次性讲清完整方案，确认后再动手
```

需要确认时，**一口气把方案说完**：要改什么、关键取舍与理由、影响面、需要用户拍板的点。确认只做这一次；之后进入实现就不再逐步请示（除非撞上红线 4 的三类例外）。

### 第 3 步：一口气实现

- 连续实现到完成，参照现有代码风格与分层（Controller 只收参/校验/返回，业务进 Service，等等）。
- **替换旧逻辑时把旧写法注释保留并写"注释原因"**（框架既有风格），注释/日志用项目语言。
- 进度更新克制且具体：有实质产出（建了哪些文件、接通了哪条链路）或方向变化才说，**不发空转播报**。
- 撞上红线 4 的例外 → 停下外显，不硬做完。

### 第 4 步：完成后自检（默认不构建，声明验证等级）

按**当前 JDK / 语言版本**做静态检查（不是泛泛"看了一遍"）：

```text
语法版本兼容   如 Java 8 不支持的语法（var/switch 表达式/文本块）、目标框架 API 版本
类型与空指针   可能的 NPE、类型不匹配、装拆箱、Optional 误用
签名一致       方法重载/覆写签名、返回类型、调用处参数对得上
字段一致       DTO ↔ 实体 ↔ Mapper ↔ 前端 API ↔ 页面 五方字段名/类型对齐
依赖与引用     import 齐全、无未引入依赖、无残留调试代码
SQL 方言       增量 SQL 是否被目标库版本支持（如 MySQL 5.7 限制）
```

- 声明验证等级 **L0（静态阅读）**；发现只有编译器能暴露的风险（返回类型退化、重载二义）→ 标注并**主动申请仅 compile 的 L1**，不擅自构建。

### 第 5 步：一次性交付完整结果

一口气给出（而不是挤牙膏）：做了什么、改了哪些文件（前端/后端/SQL 分列）、需要用户人工执行的动作、验证等级与未覆盖风险、已知风险。随后衔接 `ai-handoff-doc-update` 做收尾沉淀。

## 最终反馈格式

```text
## 交付结果

### 1. 需求理解 & 方案
- 一句话需求 + 采用的实现思路（若第 2 步确认过，注明"按已确认方案"）

### 2. 改了什么
- 前端：… / 后端：… / SQL：…（分列，附文件路径）
- 复用了哪些现有能力 / 新增了什么

### 3. 影响面 & 角色口径
- 波及模块/上下游：… / 各角色能做什么：…

### 4. 自检结果
- 验证等级：L0（静态）/ L1（仅 compile，已获放行）
- 静态检查覆盖项：语法版本/类型空指针/签名/字段一致/依赖/SQL 方言
- 未覆盖风险：…（未构建/未联调意味着什么）

### 5. 需要你做的
- 执行 SQL / 分配权限 / 重启服务 等 AI 做不了的动作

### 6. 已知风险 / 后续建议
```

## 常见坑

### 坑 1：用状态播报冒充进度（本 Skill 的起因）
"继续推进/这就去做/不再问你"连发五次也没推进多少。这些话是打断，不是工作。要么在思考、要么在编辑，别在表决心。

### 坑 2：把"确认一次"做成两个极端
每一步都问"行不行"= 绕弯子；从不确认、闷头把高风险改动做完 = 莽撞。正解：可预见的取舍**一次性**确认，不可预见的高风险例外才停。

### 坑 3：为了"不空转"冲过高风险动作
不空转 ≠ 不刹车。改表、删数据、改公共方法、动权限，即使在"一口气实现"途中也要停下确认（红线 4）。

### 坑 4：自检只说"静态看过了"，漏掉版本兼容
在 Java 8 项目里用了 Java 11 语法、在 MySQL 5.7 上写了不兼容 SQL——静态检查必须**对齐当前 JDK/库版本**，不是笼统扫一眼。

### 坑 5：跳过"想清楚"直接开写
绕弯子的真正根因。没读懂现状、没想清影响面就动手，必然边写边改方向、反复回炉。第 1 步是省时间的，不是耽误时间的。
