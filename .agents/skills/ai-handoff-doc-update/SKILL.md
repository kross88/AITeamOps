# AI Handoff Doc Update（任务后文档沉淀）

把一次任务里真实产生的项目理解，沉淀到正确的文档位置，让下一次会话（无论 Claude、Codex 还是别的工具）能快速接手。核心是**分流归位**，不是把什么都往一个文件里堆。

## 最重要的认知边界

- **AGENTS.md 是「索引 + 稳定规则」，不是任务流水账。** 每次任务都往里写总结，会让它越来越长、越来越费 token，违背它的定位。任务历史有专门的去处。
- **沉淀 ≠ 编故事。** 只记录本次任务真实产生、且对将来有用的理解。没产生新理解就如实说「本次无需沉淀」，不要为了写而写。
- **文档反映的是「写入时的事实」。** 如果旧文档与当前代码冲突，以当前代码为准，并顺手订正过期内容，不盲目相信旧文档。
- **本 Skill 不写业务代码。** 只整理文档。

---

## 红线

1. **不把任务流水堆进 AGENTS.md。** 单次任务记录一律进 `docs/ai-memory/task-log.md`。AGENTS.md 只在「稳定索引」变化时才动（见下表）。
2. **不臆造项目理解。** 没有依据的"业务规则""风险点"不准写。拿不准的标注「待确认」。
3. **不覆盖、不删除已有文档内容。** 只做「增量追加」或「精准订正某一处」，不整篇重写、不大范围格式化。
4. **改 AGENTS.md 里的稳定规则/红线前必须先问人。** 那是团队共同约束，影响所有人和所有 AI 工具，不能自行改写。索引类（模块清单、关键表、风险点路径）可直接增量更新。
5. **作用域别串味——不把用户个人事实写进项目记忆。** 用户的身份、时区、代词、个人工作习惯属**用户作用域**，归用户全局文件，**不写进项目 `docs/ai-memory`**（那是团队共享的项目记忆，掺进一个人的个人信息会污染所有人）。`docs/ai-memory` 只放**项目本身**的事实。

## 触发时机：什么算「完成一个任务」

需要沉淀的（实质改动）：

```text
- 改了业务代码、SQL、接口、配置
- 定位并修复了一个 bug
- 新增/改造了一个模块或页面
- 理清了一条调用链、一处权限逻辑、一个表关系
- 发现了一个值得记下来的坑或风险点
```

不需要沉淀的（不要打扰）：

```text
- 纯问答、纯解释、纯查代码没动东西
- 改动太小且没有沉淀价值（如改个错别字、调个日志文案）
- 探索到一半、结论未定（结论定了再沉淀）
- 噪声：工具输出/报错日志、系统元数据(ID/时间戳)、回忆型提问("还记得X吗")、含糊描述、原始对话片段
```

> 写每条记忆前先过一遍：
> - **值得记 vs 绝不记**：值得记 = 可复用 + 长期有效 + 具体可执行；绝不记 = 上面那些噪声。
> - **入库三问**：① 新颖吗（已记过就别重复）② 有把握吗（没把握标「待确认」，不当事实）③ 真会用吗（用不到就不记）。
> - **原子化**：每条读起来像一条独立的事实/决策/坑，超过 ~200 字还像原话就先蒸馏成一句。
> - **时间分类（防腐烂）**：含「现在/最近/目前/本周/暂时」等时效词的，转成绝对日期或标「截至 YYYY-MM-DD」，别把动态信息当永久事实存。

## 路由表（本 Skill 的核心）

把本次产出按类别写到对应文件，**不要混在一起**：

| 你产生了什么 | 写到哪 | 是否动 AGENTS.md |
|---|---|---|
| 单次任务记录（谁、改了什么、为什么） | `docs/ai-memory/task-log.md` | 否 |
| 对项目整体理解的更新（架构、主流程、关键约束） | `docs/ai-memory/overview.md` | 否 |
| 某模块的业务理解、字段、流程 | `docs/ai-memory/modules.md` 的 `## <模块>` 小节 | 仅新模块时在索引加一行 |
| 接口入口、请求/返回字段映射 | `docs/ai-memory/interface-map.md` | 否 |
| 表结构、表关系、关键字段含义 | `docs/ai-memory/database-map.md` | 仅新增关键表时在索引加一行 |
| 授权写库的执行台账（非只读语句） | `docs/ai-memory/db-write-log.md`（由 `mysql-guarded-write` 追加） | 否 |
| 通用风险点、易踩的坑 | `docs/ai-memory/risk-points.md` | 仅新增重大风险时在索引加一行 |
| 新模块 / 新关键表 / 新 Skill / 目录结构变动 | 对应明细文件 | **是**（只更新索引那一行） |

> `overview.md` 是团队共享理解的单一入口（开工时由 `project-context-sync` 读取）。它是「快照」不是「流水」：只在项目整体理解确有更新时**精准订正对应段落**，日常任务流水仍进 `task-log.md`，别把它写成第二个 task-log。

> 上表的 `docs/ai-memory/` 结构与 `team-ai-workspace-bootstrap` 初始化出的骨架一致，直接对应即可。
> **模块理解默认一个总文件**（`modules.md`，一模块一个 `##` 小节）——文件少、好翻阅。仅当某模块段落**过大或多人频繁冲突**时才拆出 `modules/<模块>.md`，并在 `modules.md` 原小节留一行链接。
> **向后兼容**：早期 bootstrap 版本曾建过单文件 `docs/AI开发交接记录.md`（等同 `task-log.md`）或 `modules/` 目录（一模块一文件）——项目里已有哪套就继续用哪套，一切**以项目已有结构为准，不要两套并存**。

## 执行步骤

### 第 1 步：判断要不要沉淀

对照上面「触发时机」。无实质改动或无沉淀价值 → 回复「本次无需沉淀」，结束。

### 第 2 步：探测现有文档结构

先看项目里已经有什么，路由到已有的，不要新造一套：

```bash
# 找项目级 AGENTS.md（当前目录往上找）
ls AGENTS.md ../AGENTS.md 2>/dev/null
# 找记忆目录
ls -R docs/ai-memory 2>/dev/null
ls docs/AI开发交接记录.md docs/模块验收清单.md 2>/dev/null
```

```powershell
# Windows PowerShell
Get-ChildItem AGENTS.md, ..\AGENTS.md -ErrorAction SilentlyContinue
Get-ChildItem docs\ai-memory -Recurse -ErrorAction SilentlyContinue
```

- 已有 `docs/ai-memory/` → 用它。
- 只有 `docs/AI开发交接记录.md` → 任务记录写这里。
- 都没有 → **不许沉默跳过，也不擅自建整套骨架**：把本次要沉淀的内容**先整理好**，放进任务总结的「沉淀」段，并明确请用户决定「是否建 `docs/ai-memory` 骨架并写入（可用 `team-ai-workspace-bootstrap`）」。内容不丢、决定权在用户——"项目没骨架"是要暴露给用户的冲突，不是不沉淀的理由。

### 第 3 步：给本次产出分类

把这次任务的产出对照路由表分门别类。一次任务可能命中多类（如：改了代码 + 理清了一个表关系 + 发现一个坑）。

### 第 4 步：写任务记录（必做）

往 `docs/ai-memory/task-log.md`（或 `AI开发交接记录.md`）**追加一条**，用第 9 节模板。不要改动已有条目。

### 第 5 步：路由其余产出

模块理解、接口、表、风险，分别**增量写入**对应文件。同一处理论已有内容时，精准订正那一处，不整篇重写。

### 第 6 步：仅在索引变化时更新 AGENTS.md

只有出现「新模块 / 新关键表 / 新风险点 / 新 Skill / 结构变动」时，才在 AGENTS.md 对应索引区**加/改一行**。涉及稳定规则或红线的改动，先问人（红线 4）。

### 第 7 步：控制体积

`task-log.md` 会持续增长。当它过长（如超过 ~300 行或半年以上历史）时，提示用户：把陈旧条目归档到 `docs/ai-memory/archive/task-log-<年份>.md`，主文件只留近期。不自行删历史。

### 第 8 步：反馈

按第 10 节格式告诉用户：改了哪些文件、各加了什么、AGENTS.md 动没动。

## 9. 任务记录条目模板

追加到 `task-log.md`（顶部新条目在前，倒序）。条目带**可追踪 ID 与状态**（借鉴成熟记忆插件的 `LRN-YYYYMMDD-NNN` 做法），把被动笔记变成可追踪的待办：

```markdown
## [LOG-YYYYMMDD-NN] 本次任务一句话标题

- 状态：done / pending（遗留未完）
- 改动范围：（哪些文件 / 模块 / 表）
- 做了什么：（一句话说清，能让人一眼看懂）
- 为什么：（动机 / 触发的需求）
- 关键决策 / 坑：（可选，遇到的非显然问题）
- 影响面：（是否动了公共方法 / 权限 / 表结构 / 配置）
- 验证方式：（怎么确认这次是对的；未验证就写"未验证"）
- 后续动作：（可选，留下的 TODO；没有就省略）
```

示例：

```markdown
## [LOG-20260615-01] 修复入库单登记重复提交

- 状态：done
- 改动范围：service 入库模块 StockInController / StockInServiceImpl
- 做了什么：登记接口加幂等校验，按业务单号 + 操作人 5 秒内去重
- 为什么：用户快速双击导致一次入库生成两条记录
- 关键决策 / 坑：原代码无唯一约束，先用应用层去重，DB 唯一索引列入风险点待补
- 影响面：未动公共方法，未改表结构
- 验证方式：本地连点 3 次，仅生成 1 条；已有单测通过
- 后续动作：给 stock_in 加业务单号唯一索引（已记入 risk-points）
```

## 收尾沉淀结论（必须可见，缺了 = 没跑收尾）

不管本次有没有东西可沉淀，**任务总结必须包含一段「沉淀」结论**——内容是三层记忆各扫一遍的结果：

```text
沉淀：项目层——已写入 docs/ai-memory/xxx / 无骨架，内容已备好待你决定 / 本次无需沉淀（原因）；
全局层——发现 1 条够格的跨项目经验，提议：「…」（待确认） / 无；
用户层——无新的用户个人事实 / 有：…（待确认）。
```

为什么必须可见：收尾最容易被"交付物清单"挤掉——用户要的产出交完了，规范要的收尾就忘了。**「本次无需沉淀」也必须写出来**，它证明扫描发生过；缺了这段，用户一眼就能发现收尾没跑。即使在没有加载本 Skill 全文的降级场景，这三行也必须输出。

## 11. 与 git-commit-guard 的配合

推荐顺序：**先沉淀文档，再提交代码**，让文档变更和代码变更进同一个 commit，保证「代码 + 项目记忆」始终同步。

```text
完成任务  →  ai-handoff-doc-update（沉淀文档）  →  git-commit-guard（检查 + commit）
```

这样翻 git 历史时，每次代码改动都能对上一条文档沉淀。

## 12. 团队并发与合并约定

`docs/ai-memory/` 跟代码一起进 git，多人协作时**会产生 merge 冲突**——尤其多人同一天往 `task-log.md`、`database-map.md` 追加。沉淀时遵守以下约定，把冲突降到最低、即使冲突也好解：

- **永远「顶部追加」新条目，不动旧条目。** 新内容加在文件最上方（或对应小节顶部），让冲突集中在文件头部、易于人工合并。
- **条目原子化、自带日期与作者/会话标识。** 一条记录是一个完整块，不跨条目交叉编辑，冲突时各留一份即可。
- **明细按主题分文件、文件内按 `##` 小节分段**（模块在 `modules.md` 各占一节、一表一段）——各改各的小节，git 对同文件不同区域通常能自动合并；某模块小节**频繁冲突或过大**时再拆独立文件。
- **`overview.md` 谨慎改、改前先 `project-context-sync` 同步远程**，只精准订正变化的段落，避免整段重写引发大块冲突。
- **遇到 ai-memory 的合并冲突**：内容类文档通常「两边都保留」即正确（两条任务记录都留），不要随手删掉别人那一份。拿不准交人。

> 一句话：把记忆文件写成「只增不改、顶部追加、原子条目」的流水/快照，天然对并发友好。

### 新内容撞上旧内容时的 7 种处理（借鉴成熟记忆插件的去重词汇）

不要一律「追加」或一律「覆盖」，按下面判：

```text
CREATE   全新信息，旧记忆没覆盖到        → 新建
SKIP     重复，或信息比旧条目更少（退化）→ 跳过，绝不用更弱的覆盖更全的
MERGE    给旧条目补充了新细节            → 合并成一条
SUPERSEDE 同一可变事实变了（如表结构/约定改了）→ 旧条目标「已过时(历史)」、另建当前条，不删历史
SUPPORT  在某情境下印证了旧结论          → 旧条目下补一句佐证
CONTEXTUALIZE 同一主题、不同情境的细化   → 作为情境补充，不否定旧的
CONTRADICT 直接冲突                      → 两条都留、各自标注情境/时间，交人裁决
```

### 哪些文件「只追加」、哪些「可版本替换」

- **只追加（append-only）**：`task-log.md`、个案/事件类记录——发生过就是历史，永远 CREATE/SKIP，不改旧条目。
- **可版本替换（temporal-versioned）**：`overview.md`、`database-map.md`、`interface-map.md` 里的「当前事实」——事实变了用 **SUPERSEDE**（旧的标历史、写新的当前），别让过期描述和新描述并列误导。

## 13. 记忆书写约定：L0 / L1 / L2 三层 + 热冷分层

**每条记忆写三层**（借鉴成熟记忆插件的 abstract/overview/content），既能快速扫、又能深挖，顺手解决「该写多详细」：

```text
L0 一句话索引   → 进 MEMORY 索引 / overview.md 的要点行（必写，最精炼）
L1 结构化要点   → 该条目的小标题 + 要点（modules/database-map 等明细的主体）
L2 完整叙述     → 背景、过程、根因等细节（个案/复杂模块才需要）
```

**热冷分层**（借鉴 core/working/peripheral，但人工维护、不追踪访问数）：

```text
core(始终加载，保持精简)   → overview.md + AGENTS.md 索引：项目心智模型，开工必读
working(活跃)              → 近期 task-log、当前在改模块的 modules.md 小节
peripheral(老化/归档)      → archive/ 下的旧 task-log
```

- 高频反复用到的事实，**手动上提**到 `overview.md`（core）；陈旧的下沉到 `archive/`。
- core 层越小越好——它每次开工都被 `project-context-sync` 读，臃肿就拖慢所有人。

**关系链接（让记忆不再是孤岛）**：

ai-memory 各文件按主题分仓，但业务是相互关联的——"这个模块用了这张表、由这个接口暴露"。写记忆时用 `[[名称]]` 把关联标出来：

```text
指向约定（纯文本，零运行时，统一为「总文件 + ## 小节」）：
  [[某模块]]   → modules.md 里的「## 某模块」小节（已拆分的 → modules/某模块.md）
  [[某表名]]   → database-map.md 里的「## 某表名」小节
  [[某接口]]   → interface-map.md 里的对应小节
```

- **写模块理解时**，涉及的关键表、上下游模块、对外接口用 `[[]]` 标注；**overview.md 的模块清单**每行链接到 `[[模块]]`——它因此成为整个记忆网的关系中枢。
- **`[[]]` 指向尚未建档的实体不是错误**，恰是"值得建档"的标记——下次有人碰到它，就知道该补一份。
- 链接的回报在读侧：`project-context-sync` 只需读 overview + 本次相关的一两个文件，顺着链接就能精准扩展，不必全量加载。

## 常见坑

### 坑 1：把任务总结写进 AGENTS.md
最常见的错误。AGENTS.md 会被每次会话加载，堆任务流水 = 持续烧 token。任务记录永远进 `task-log.md`。

### 坑 2：为了写而写
没产生新理解还硬凑总结，制造噪音文档。无价值就如实说「本次无需沉淀」。

### 坑 3：整篇重写已有文档
只增量追加 / 精准订正。整篇重写既丢历史，又容易把别人写的内容覆盖掉。

### 坑 4：两套文档结构并存
项目已有 `AI开发交接记录.md` 就别再造 `task-log.md`，反之亦然。先探测，再归位。

### 坑 5：擅自改 AGENTS.md 的红线/规则
索引可以增量加，规则和红线是团队约束，改前必须问人。

### 坑 6：task-log 无限膨胀
不归档会越读越慢。过长时提示归档，但不自行删历史。
