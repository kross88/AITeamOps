# AI Handoff Doc Update（任务后文档沉淀）

把一次任务里真实产生的项目理解，沉淀到正确的文档位置，让下一次会话（无论 Claude、Codex 还是别的工具）能快速接手。核心是**分流归位**，不是把什么都往一个文件里堆。

## 最重要的认知边界

- **AGENTS.md 是「索引 + 稳定规则」，不是任务流水账。** 每次任务都往里写总结，会让它越来越长、越来越费 token，违背它的定位。任务历史有专门的去处。
- **沉淀 ≠ 编故事。** 只记录本次任务真实产生、且对将来有用的理解。没产生新理解就如实说「本次无需沉淀」，不要为了写而写。
- **文档反映的是「写入时的事实」。** 如果旧文档与当前代码冲突，以当前代码为准，并顺手订正过期内容，不盲目相信旧文档。
- **本 Skill 不写业务代码。** 只整理文档。

---

## 红线

1. **不把任务流水堆进 AGENTS.md。** 单次任务记录一律进 `docs/ai-memory/task-log.md`。AGENTS.md 只在「稳定索引」变化时才动（见下表）。
2. **不臆造项目理解。** 没有依据的"业务规则""风险点"不准写。拿不准的标注「待确认」。
3. **不覆盖、不删除已有文档内容。** 只做「增量追加」或「精准订正某一处」，不整篇重写、不大范围格式化。
4. **改 AGENTS.md 里的稳定规则/红线前必须先问人。** 那是团队共同约束，影响所有人和所有 AI 工具，不能自行改写。索引类（模块清单、关键表、风险点路径）可直接增量更新。

## 触发时机：什么算「完成一个任务」

需要沉淀的（实质改动）：

```text
- 改了业务代码、SQL、接口、配置
- 定位并修复了一个 bug
- 新增/改造了一个模块或页面
- 理清了一条调用链、一处权限逻辑、一个表关系
- 发现了一个值得记下来的坑或风险点
```

不需要沉淀的（不要打扰）：

```text
- 纯问答、纯解释、纯查代码没动东西
- 改动太小且没有沉淀价值（如改个错别字、调个日志文案）
- 探索到一半、结论未定（结论定了再沉淀）
```

## 路由表（本 Skill 的核心）

把本次产出按类别写到对应文件，**不要混在一起**：

| 你产生了什么 | 写到哪 | 是否动 AGENTS.md |
|---|---|---|
| 单次任务记录（谁、改了什么、为什么） | `docs/ai-memory/task-log.md` | 否 |
| 某模块的业务理解、字段、流程 | `docs/ai-memory/modules/<模块>.md` | 仅新模块时在索引加一行 |
| 接口入口、请求/返回字段映射 | `docs/ai-memory/interface-map.md` | 否 |
| 表结构、表关系、关键字段含义 | `docs/ai-memory/database-map.md` | 仅新增关键表时在索引加一行 |
| 通用风险点、易踩的坑 | `docs/ai-memory/risk-points.md` | 仅新增重大风险时在索引加一行 |
| 新模块 / 新关键表 / 新 Skill / 目录结构变动 | 对应明细文件 | **是**（只更新索引那一行） |

> 上表的 `docs/ai-memory/` 结构与 `team-ai-workspace-bootstrap` 初始化出的骨架一致，直接对应即可。
> **向后兼容**：早期 bootstrap 版本曾建过单文件 `docs/AI开发交接记录.md`，若项目里已有它，就继续把「任务记录」写进去，作用等同 `task-log.md`，不要再另起 `task-log.md`。一切**以项目已有结构为准，不要两套并存**。

## 执行步骤

### 第 1 步：判断要不要沉淀

对照上面「触发时机」。无实质改动或无沉淀价值 → 回复「本次无需沉淀」，结束。

### 第 2 步：探测现有文档结构

先看项目里已经有什么，路由到已有的，不要新造一套：

```bash
# 找项目级 AGENTS.md（当前目录往上找）
ls AGENTS.md ../AGENTS.md 2>/dev/null
# 找记忆目录
ls -R docs/ai-memory 2>/dev/null
ls docs/AI开发交接记录.md docs/模块验收清单.md 2>/dev/null
```

```powershell
# Windows PowerShell
Get-ChildItem AGENTS.md, ..\AGENTS.md -ErrorAction SilentlyContinue
Get-ChildItem docs\ai-memory -Recurse -ErrorAction SilentlyContinue
```

- 已有 `docs/ai-memory/` → 用它。
- 只有 `docs/AI开发交接记录.md` → 任务记录写这里。
- 都没有 → 提示用户「建议先用 `team-ai-workspace-bootstrap` 建文档骨架」，不要擅自创建整套结构。

### 第 3 步：给本次产出分类

把这次任务的产出对照路由表分门别类。一次任务可能命中多类（如：改了代码 + 理清了一个表关系 + 发现一个坑）。

### 第 4 步：写任务记录（必做）

往 `docs/ai-memory/task-log.md`（或 `AI开发交接记录.md`）**追加一条**，用第 9 节模板。不要改动已有条目。

### 第 5 步：路由其余产出

模块理解、接口、表、风险，分别**增量写入**对应文件。同一处理论已有内容时，精准订正那一处，不整篇重写。

### 第 6 步：仅在索引变化时更新 AGENTS.md

只有出现「新模块 / 新关键表 / 新风险点 / 新 Skill / 结构变动」时，才在 AGENTS.md 对应索引区**加/改一行**。涉及稳定规则或红线的改动，先问人（红线 4）。

### 第 7 步：控制体积

`task-log.md` 会持续增长。当它过长（如超过 ~300 行或半年以上历史）时，提示用户：把陈旧条目归档到 `docs/ai-memory/archive/task-log-<年份>.md`，主文件只留近期。不自行删历史。

### 第 8 步：反馈

按第 10 节格式告诉用户：改了哪些文件、各加了什么、AGENTS.md 动没动。

## 9. 任务记录条目模板

追加到 `task-log.md`（顶部新条目在前，倒序）：

```markdown
## YYYY-MM-DD 本次任务一句话标题

- 改动范围：（哪些文件 / 模块 / 表）
- 做了什么：（一句话说清，能让人一眼看懂）
- 为什么：（动机 / 触发的需求）
- 关键决策 / 坑：（可选，遇到的非显然问题）
- 影响面：（是否动了公共方法 / 权限 / 表结构 / 配置）
- 验证方式：（怎么确认这次是对的；未验证就写"未验证"）
```

示例：

```markdown
## 2026-06-15 修复入库单登记重复提交

- 改动范围：service 入库模块 StockInController / StockInServiceImpl
- 做了什么：登记接口加幂等校验，按业务单号 + 操作人 5 秒内去重
- 为什么：用户快速双击导致一次入库生成两条记录
- 关键决策 / 坑：原代码无唯一约束，先用应用层去重，DB 唯一索引列入风险点待补
- 影响面：未动公共方法，未改表结构
- 验证方式：本地连点 3 次，仅生成 1 条；已有单测通过
```

## 11. 与 git-commit-guard 的配合

推荐顺序：**先沉淀文档，再提交代码**，让文档变更和代码变更进同一个 commit，保证「代码 + 项目记忆」始终同步。

```text
完成任务  →  ai-handoff-doc-update（沉淀文档）  →  git-commit-guard（检查 + commit）
```

这样翻 git 历史时，每次代码改动都能对上一条文档沉淀。

## 常见坑

### 坑 1：把任务总结写进 AGENTS.md
最常见的错误。AGENTS.md 会被每次会话加载，堆任务流水 = 持续烧 token。任务记录永远进 `task-log.md`。

### 坑 2：为了写而写
没产生新理解还硬凑总结，制造噪音文档。无价值就如实说「本次无需沉淀」。

### 坑 3：整篇重写已有文档
只增量追加 / 精准订正。整篇重写既丢历史，又容易把别人写的内容覆盖掉。

### 坑 4：两套文档结构并存
项目已有 `AI开发交接记录.md` 就别再造 `task-log.md`，反之亦然。先探测，再归位。

### 坑 5：擅自改 AGENTS.md 的红线/规则
索引可以增量加，规则和红线是团队约束，改前必须问人。

### 坑 6：task-log 无限膨胀
不归档会越读越慢。过长时提示归档，但不自行删历史。
