# MySQL Guarded Write（授权下的谨慎写库）

用户明确授权后，代替用户执行写库 SQL——但**授权不等于放开手脚**：范围限定在本次需求相关的表，每一条都预检、备份、执行、回读、报告。本 Skill 是 `mysql-readonly-probe-via-java` 的**写侧姊妹**：环境准备（JDBC via JDK + connector jar）完全复用它，只是解锁了受控的写能力。

## 最重要的认知边界

- **写库不可逆。** UPDATE/DELETE 跑错一条，没有 Ctrl+Z。所以每条语句都要有"跑之前知道会发生什么、跑之后确认发生了什么、出错了怎么回去"三件套。
- **授权是一次性的、范围限定的。** 用户说"执行这个 SQL"授权的是**这个 SQL 和这次需求**，不是"以后都可以写库"，更不是"顺手把别的表也改了"。
- **先确认连的是哪个库。** Spring Boot 多 profile 下连错库（把测试语句跑到生产）是最灾难的错误。沿用 readonly-probe 第 0 步：先确认激活 profile 与最终 datasource。

## 红线

1. **无明确授权绝不写库。** 授权必须是**本次会话内**用户对**具体写操作**的明确表态（"你执行吧""跑这个 SQL"）。"帮我看看数据"不是授权；上次会话的授权不延续。未授权时走默认路径：出 SQL 给用户 + 说明风险，或只读探测。
2. **只操作需求相关表，白名单先过目。** 动手前列出本次要触碰的表清单给用户看；执行中发现需要动白名单外的表 → 停下再确认，授权不扩散。
3. **绝对禁止（即使用户说"随便执行"）**：
   ```sql
   DROP DATABASE ...          -- 库级销毁
   TRUNCATE ...               -- 不可回滚的清表
   UPDATE/DELETE 无 WHERE     -- 全表写操作
   GRANT / REVOKE ...         -- 权限变更
   SET GLOBAL / SET PERSIST   -- 服务器级状态
   ```
   这类操作只能把语句写给用户，由用户亲自执行。
4. **逐条报告，先预检后执行。** 每条 SQL：执行前报告「语句 + 目标表 + 预期影响行数」；执行后报告「实际影响行数 + 回读验证结果」。实际与预期明显不符（如预估 3 行实际 300 行）→ 立即停止后续语句，报告用户。
5. **可回滚是前提。** UPDATE/DELETE 前先 `SELECT` 保存受影响行的当前值（贴进总结或存备份表）；DDL 前记录原表结构（`SHOW CREATE TABLE`）；每条写操作给出对应的**回滚 SQL**。做不到回滚准备的操作，交回用户。
6. **高危语句逐条再确认。** `DROP TABLE`、`ALTER` 删列/改类型、影响行数 > 100 的 UPDATE/DELETE——即使已有总体授权，执行前也要单独跟用户确认这一条。
7. **凡写操作必落执行台账（可回溯是硬要求）。**
   - **SELECT 等只读语句不记**（预检采样、回读验证的 SELECT 都不入台账，避免噪声）。
   - **凡执行 INSERT / UPDATE / DELETE / DDL 等非只读语句，每一条都必须记入执行台账**，缺一条即视为违规。
   - 新结构项目的台账路径是 `docs/ai-memory/task-log.md`（`DBW-` 条目，追加不覆盖）。早期项目若已经使用 `docs/ai-memory/db-write-log.md`，继续写原文件，不自动迁移、也不向两处双写。项目两处都没有时，先把台账内容备好、在收尾里提请用户决定是否创建 `task-log.md`，不因缺骨架就不记（对齐 `ai-handoff-doc-update` 的兜底）。
   - 每条台账至少含：执行时间、授权依据（用户原话）、目标库、SQL 原文、预期/实际影响行数、回滚 SQL。格式见文末「执行台账条目」。
   - 台账是"已执行事实"的账本，最终报告是"本次汇总"，两者都要有。

## 执行流程

### 第 0 步：确认授权与范围
- 复述用户的授权内容（哪些 SQL / 哪个需求）。
- 列出将要触碰的**表白名单**与操作类型，给用户过目。
- 确认目标库：激活 profile、最终 `spring.datasource.url`（复用 readonly-probe 第 0 步，**绝不只看 application.yml**）。

### 第 1 步：环境准备
复用 `mysql-readonly-probe-via-java` 的方案：找 JDK、找 mysql-connector jar、临时目录写 Java、无 BOM UTF-8、classpath 分隔符按平台。写操作代码同样用完即删（含明文密码）。

### 第 2 步：逐条预检
```sql
-- UPDATE/DELETE：先用同 WHERE 的 SELECT 预估并备份
SELECT COUNT(*) FROM t WHERE <同条件>;          -- 预期影响行数
SELECT * FROM t WHERE <同条件>;                 -- 备份受影响行（留存到总结）
-- DDL：先留底
SHOW CREATE TABLE t;
-- 大表操作：EXPLAIN 确认走索引，避免锁全表
```

### 第 3 步：执行
- 多条语句能包事务就包事务（`setAutoCommit(false)`），任一失败整体回滚。
- DDL（MySQL 隐式提交、不进事务）逐条执行，每条之间回读确认。
- 每条执行后立即取 `affected rows`，与预检对比。

### 第 4 步：回读验证 + 落台账
- 执行后用 SELECT 回读关键数据，确认结果符合意图（不只是"语句成功"，而是"数据变成了想要的样子"）。
- **每条写语句执行完，立即以 `DBW-` 条目追加到项目当前采用的唯一台账文件**：新结构用 `task-log.md`，已有旧结构用 `db-write-log.md`（红线 7）。SELECT 回读/预检不记。

### 第 5 步：报告 + 沉淀
汇总最终报告（下方格式），并确认台账已逐条落盘、回滚 SQL 齐备。

## 执行台账条目（追加到项目当前唯一台账文件，一条写操作一条）

```markdown
## [DBW-YYYYMMDD-NN] 一句话说明本次改了什么
- 执行时间：YYYY-MM-DD HH:MM
- 授权依据：用户原话「…」
- 目标库：host / database（profile 来源，已确认非误连）
- SQL：`UPDATE t SET ... WHERE ...`
- 预期影响：n 行；实际影响：n 行（一致 / 不一致已中止）
- 回滚 SQL：`UPDATE t SET ... WHERE ...`（或备份表/CSV 位置）
- 回读验证：通过 / …
```

> 台账**只记非只读执行**（INSERT/UPDATE/DELETE/DDL）。SELECT 无论多少条都不进台账。
> 归档含 DBW 的旧 task-log 前必须取得用户确认，并在当前 `task-log.md` 保留指向归档文件的固定索引，确保审计记录可定位。

## 最终报告格式

```text
## 写库操作报告

### 0. 授权依据
- 用户原话/时间：…；授权范围：…

### 1. 目标库
- host/database/profile 来源：…（已确认非误连）

### 2. 表白名单
- 涉及表：…（均属本次需求）

### 3. 逐条执行记录
| # | SQL 摘要 | 预期影响 | 实际影响 | 回读验证 | 回滚 SQL |

### 4. 备份留存
- 受影响行备份：…（贴录/备份表名）

### 5. 异常与中止
- 无 / 第 n 条实际影响与预期不符已中止：…

### 6. 沉淀
- 执行台账与任务记录：已逐条以 DBW- 条目追加到项目当前唯一台账文件（task-log.md / 旧项目 db-write-log.md；本次 n 条写操作；缺骨架时内容已备好，提请用户决定）
```

## 与 mysql-readonly-probe-via-java 的分工

```text
默认 / 排查 / 核验      → readonly-probe（绝不写库，这条红线永远不变）
用户明确授权的写操作     → 本 Skill（受控写，逐条报告可回滚）
本 Skill 红线禁止的操作  → 写 SQL 给用户，本人执行
```

readonly-probe 的「绝不写库」不因本 Skill 存在而松动——**遇到写需求不是在探测里破例，而是显式切换到本 Skill 并重新确认授权**。

## 常见坑

### 坑 1：连错库
多 profile 时把语句跑进生产库。第 0 步必须确认最终 datasource，报告里写明目标库。

### 坑 2：把一次授权当长期授权
上个任务用户让执行过 SQL ≠ 这个任务也可以。每次写操作重新确认。

### 坑 3：授权扩散
执行中发现"顺手改一下另一张表就好了"——停。白名单外的表需要重新确认。

### 坑 4：只看"执行成功"不看影响行数
`UPDATE ... 0 rows affected` 也是"成功"。必须对比预期/实际行数 + 回读数据。

### 坑 5：没备份就跑
备份受影响行是执行的前置条件，不是可选项。没备份 = 没有回滚能力 = 交回用户。
