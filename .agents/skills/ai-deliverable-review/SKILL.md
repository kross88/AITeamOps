# AI Deliverable Review（AI 交付物验收级复核）

复核另一个 AI 或同事交付的功能模块。目标**不是证明"做完了"，而是主动找出哪里不好、哪里不完整、哪里上线后会出问题**——找到确定性问题就地修复，争议问题按安全口径处理并声明。

## 最重要的认知边界

- **交付方的总结不是证据。** "Codex 说做了 X" ≠ 代码里真的有 X。每一项都要在代码/SQL 里亲自核实。
- **静态核对 ≠ 编译验证。** 链式调用返回类型退化、方法重载二义性、过时 API——这类问题只有编译器能暴露，读代码读不出来。所以必须**声明验证等级**，并主动申请最小验证动作。
- **前端隐藏不算权限。** 按钮不显示挡不住直接调接口，权限必须后端兜底。
- **复核者也会引入 bug。** 自己修改的部分同样要过验证，不因"我是来复核的"就豁免。

## 红线

1. **不信总结，一切以代码为准。** 交付方列的清单只当作核对索引，不当作事实。
2. **遵守环境禁令，但主动申请最小验证。** 用户未允许时不 build/不写库/不启动服务；同时应主动提出「是否允许仅 compile（不 package、不启动）」——那是暴露编译期问题的最低成本手段。
3. **只直接修确定性问题。** 明显 bug、字段不一致、权限缺失、状态流转错误可直接修；**业务口径争议不擅自改逻辑**，按「更安全、更符合上线验收」的口径处理，并在总结里声明处理口径（冲突外显，不沉默选边）。
4. **如实全报。** 修不了的、没把握的、没验证到的，一律写进总结，不粉饰。
5. **数据库探测只读。** 对照真实库用 `mysql-readonly-probe-via-java`，绝不写库。

## 验证等级（必须在结论顶部声明）

```text
L0 静态阅读   只读了代码与 SQL（默认起点）
L1 编译通过   后端 compile / 前端单文件编译或类型检查
L2 测试通过   单元测试/集成测试跑过
L3 运行联调   启动服务、真实角色实测
```

- 结论顶部写明「本次复核达到 Lx，未达到的等级意味着什么风险」。
- 被禁止构建时，L0 就是天花板——**必须显式说明**"存在编译期/运行时问题无法暴露的风险"，并申请放开 L1。

## 执行流程

### 第 0 步：开工声明
按 `project-context-sync`：同步远程（**待提交区有交付方改动时跳过 pull 并声明**）、加载 `docs/ai-memory`、输出 2-3 行开工声明。

### 第 1 步：圈定改动范围
```bash
git status --porcelain      # 待提交区改动
git diff --stat             # 或与基线分支的 diff
```
列出前端/后端/SQL/文档四类文件清单，作为后续核对的地图。交付方口头列的入口/文件，逐一验证真实存在。

### 第 2 步：SQL 层核对
- **数据库版本兼容**：语法是否被目标库版本支持（如 MySQL 5.7 不支持的语法、生成列/函数索引限制）。
- **与现有结构一致**：菜单表、权限表、字典表字段是否和现有系统一致；外键/ID 类型与关联表匹配（如 varchar 用户 ID）。
- **幂等可重复执行**：重复跑不会因已存在记录而失败。
- **约束合理**：唯一约束、索引、逻辑删除标志位。
- **用 `mysql-readonly-probe-via-java` 对照真实库**：确认目标库版本、现有表结构、SQL 是否已执行过——这是交付方通常不做的验证。
- **字段格式假设核验**：接口代码对字段的假设（逗号分隔 vs JSON、字典值域、ID 类型 varchar/bigint、日期格式）用 SELECT 采样对照真实数据——"代码看起来对但格式假设错了"是最隐蔽的一类交付缺陷。

### 第 3 步：后端层核对
- **权限后端兜底**（逐条查注解 + 业务归属校验），典型四类越权：
  ```text
  ① 归属越权：A 不能操作 B 的数据（owner 字段校验）
  ② 未授权数据访问：无资格用户直接调接口拿敏感数据
  ③ 自审：审核人不能审核自己提交的事项
  ④ 状态绕过：下架/禁用的数据不能绕过列表直接按 ID 访问
  ```
- **敏感字段隔离**：普通视角的 VO 是否根本不含敏感字段（而不是返回后前端不显示）。
- **状态流转闭环**：每个动作（新增/修改/上架/下架/删除/申请/取消/审核）后各表状态一致；驳回后原生效数据不变。
- **并发防护**：两个操作者同时处理同一事项（如双审核员、取消 vs 通过）——用状态条件更新抢占，不裸 update。
- **重复提交拦截**：审核中不能再次提交；同人同对象不能重复申请（应用层 + 数据库唯一约束双保险）。
- **删除安全**：逻辑删除、审核中禁删、历史记录不断链。

### 第 4 步：前端层核对
- **页面真的可用**：菜单/路由/组件路径对得上，动态菜单能加载。
- **搜索真实生效**：每个筛选条件前后端字段名一致、真的传参、后端真的过滤。
- **字段一致性**：VO/DO/Mapper/前端 API 类型/页面字段五方对齐。
- **日期兼容**：展示兼容 string/number/数组格式；日期范围搜索遵循项目惯例（如 `:default-time`，否则结束日当天查不到）。
- **按钮显示条件**：与状态机一致；**展示形态符合需求**（如要求卡片式就不能只交表格）。

### 第 5 步：需求清单逐项核对 + 角色隔离矩阵
- 原始需求逐条打钩/打叉，不允许"大体都有"。
- 每类角色横向过一遍：能看什么、能做什么、不能做什么，是否与口径一致。

### 第 6 步：修复
- 确定性问题直接修，修完后**自己的改动也要过第 7 步验证**。
- 争议问题按安全口径处理 + 总结声明；需要新 SQL 时**新增增量 SQL，不改已执行过的旧 SQL**。

### 第 7 步：验证（能到几级到几级）
按环境允许的上限做验证，并如实记录等级。被禁止时申请放开 L1（仅 compile）。

### 第 8 步：输出复核总结 + 收尾沉淀
生成/更新复核总结文档（模板见下），并按 `ai-handoff-doc-update` 做收尾三层扫描（项目层/全局层/用户层），总结里带「沉淀」段。

## 复核总结模板

```text
# <模块名> 复核总结

## 0. 验证等级
- 本次达到：L_（含义与未覆盖风险：…）

## 1. 检查范围
- 前端文件：… / 后端文件：… / SQL：… / 数据库探测：…

## 2. 发现的问题（按严重度）
| 严重度 | 问题 | 后果 | 状态(已修复/建议后续) |

## 3. 安全口径确认（均需后端兜底证据）
- 归属越权 / 未授权访问 / 自审 / 状态绕过 / 敏感字段隔离 / 重复提交 / 并发

## 4. 需求清单逐项核对
- 第 n 项：满足 / 不满足（差什么）

## 5. 角色隔离结论
## 6. 争议口径声明（冲突点 + 我选了哪边 + 为什么）
## 7. 需要用户执行的动作（SQL / 授权配置 / 联调验收清单）
## 8. 沉淀（三层扫描结论）
```

## 常见坑

### 坑 1：把交付方总结当事实
"总结说有审核闭环"→ 必须找到审核状态回写的代码才算数。

### 坑 2：静态核对后宣布"理论无障碍"
签名逐一核对过依然可能编不过（返回类型退化、重载二义性）。老实标 L0，申请 L1。

### 坑 3：权限只看了前端
v-if 隐藏按钮不拦 curl。逐接口查注解 + 归属校验。

### 坑 4：SQL 只看语法不对库
语法对但目标库版本不支持 / 字段和现有表对不上。用只读探测对照真实库。

### 坑 5：复核改出新问题
自己修的代码没验证。修复后的改动与原交付同等对待，一起过验证等级。

### 坑 6：争议口径沉默选边
按安全口径处理没问题，但必须在总结第 6 节声明，否则用户不问就永远不知道。
