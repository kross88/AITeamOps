# 需求说明模板（给 AI 的开发任务书）

> **为什么需要这个模板**：同一个 AI，拿到一句「帮我加个功能」和拿到一份结构化任务书，交付质量天差地别。本模板提炼自真实项目中验证有效的写法——把「口径、边界、默认决策、验收标准」一次性交代清楚，AI 就能一口气完成完整闭环而不必中途反复问。写好的需求说明归档到项目 `docs/requirements/`，随代码进 git，后续复核、接手的 AI 都能读到原始需求；目录在出现首份需要独立归档的需求时再创建，不预建空目录。
>
> 使用方式：复制本模板 → 按注释填写 → 删掉不适用的节和所有「说明：」行。不是每个需求都要全部节；小需求至少保留 0、1、8、9 节。

---

# <模块/功能名> 功能说明

## 0. 执行规则与环境边界

说明：AI 行动边界（能否 build/测试/连库）应写在项目 AGENTS.md 里一劳永逸；这里只写本次的特例与提醒。

1. 先阅读现有项目结构与既有写法（路由、权限、字典、SQL 增量文件、公共组件），优先复用，不重复造轮子。
2. 涉及数据库的字段/菜单/字典，先查现有表结构与真实数据（只读探测），不凭空假设。
3. 遇到未明确的细节，按第 9 节「默认决策口径」自主处理，不要中途停下提问；处理口径在最终总结中说明。
4. 本次特批/特禁：（如"允许 compile"/"禁止一切构建"——没有就删）
5. 新增 SQL 一律增量文件，不修改已执行过的旧 SQL。
6. 不做大范围格式化、不改无关模块与 import。

## 1. 模块定位与业务闭环

说明：一段话讲清这个功能给谁用、解决什么问题；再用一条链路把完整业务闭环画出来——AI 对"闭环"的理解质量直接决定交付质量。

```text
（示例）角色A 维护资料 → 提交审核 → 角色B 审核 → 通过后角色C 可见可申请
      → 申请审核通过后可见敏感信息 → 支持撤销/驳回重提/记录追溯
```

## 2. 需求清单（逐条编号）

说明：逐条编号是为了让 AI 的总结和复核能"逐项打钩"，防止"大体都做了"。

| 编号 | 功能 | 关键说明 |
|---|---|---|
| 1 | 快速搜索 | 支持哪些维度；前后端字段必须一致、后端真实过滤 |
| 2 | … | … |

## 3. 角色与权限口径

说明：有权限隔离的功能必写。每类角色写「可用能力 + 限制」，限制往往比能力更重要（不能看什么、不能操作谁的数据、不能审自己的）。**并注明：权限必须后端兜底，前端隐藏不算权限。**

### 角色A（如：管理员）
- 可以：…
- 不可以：…

## 4. 页面与交互要求

说明：只写「必须怎样」的硬要求（如"用户侧必须卡片式，管理侧用表格"「表单 label 带中文冒号」「删除需二次确认」），样式细节让 AI 沿用现有系统风格。

## 5. 状态与流转

说明：状态字段分开设计（审核状态/上下架状态/申请状态/删除状态不混在一个字段），每个状态机列出合法流转与禁止项（审核中不可重复提交、驳回可重提等）。

## 6. 数据与接口要求

说明：表名/字段给「业务建议」，最终以项目现有规范为准；明确哪些复用现有能力（附件/字典/操作日志/审核流），没有通用能力才新增最小闭环表。敏感字段（调用地址/密钥/手机号等）单列，写明谁可见。

## 7. 校验与安全要求

1. 前端 rules 与后端 VO 双重必填校验。
2. 权限后端兜底：归属校验（不能操作别人的数据）、状态绕过拦截（下架/删除数据不可按 ID 直访）、自审拦截。
3. 重复提交拦截：应用层 + 数据库唯一约束双保险。
4. 关键操作加事务；并发场景用状态条件更新抢占。

## 8. 交付物要求

1. 新增/修改文件清单（前端、后端、SQL 分列）。
2. 增量 SQL 文件路径 + 需要人工执行的动作清单（执行 SQL、分配权限等 AI 做不了的事）。
3. 开发总结文档：`docs/<模块名>_开发总结.md`（含状态流转、角色权限、自测清单）。
4. **验证等级声明**：达到 L0/L1/L2/L3 哪级、未达级别的风险。
5. 自测清单：按角色 × 场景列出验收点（正向 + 越权/绕过反向用例）。

## 9. 默认决策口径

说明：这一节是「不要中途问我」能成立的前提——把可能有歧义的点提前拍板。写不全没关系，AI 会按"更安全、更符合上线验收"的口径处理并在总结中声明。

1. （示例）审核通过后默认进入「已通过+未上架」，再手动上架。
2. （示例）有通用审核流就复用，没有就做模块内最小审核闭环。
3. （示例）删除一律逻辑删除；审核中禁止删除。
4. 其余未明确处：按更安全、可追溯、符合验收的方式处理，并在总结中说明口径。

## 10. 最终回复格式

说明：固定回复格式让多个 AI（开发的、复核的）产出可对照。可直接要求按下面骨架回复。

```text
一、已完成内容
二、新增/修改前端文件
三、新增/修改后端文件
四、新增 SQL（路径/表/菜单/字典/索引）
五、需要人工执行的动作
六、角色权限实现情况（逐角色）
七、核心流程实现情况（逐流程）
八、验证等级与未执行事项
九、已知风险 / 后续建议
```
